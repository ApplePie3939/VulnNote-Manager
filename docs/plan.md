# 開発計画（plan.md）

このドキュメントは `docs/memo.txt` の要望と、`docs/grill-me.md` の指示に基づき
ユーザーとの一問一答を通じて合意した内容をもとに作成した開発計画である。

参照：`docs/memo.txt`、`docs/grill-me.md`、`AGENTS.md`

## 1. 目的

ウェブサイトの脆弱性診断中に記録したメモを整理し、
再現手順・影響・証跡・対策案を分かりやすくまとめるためのツールを作る。

散らばりがちな診断メモを統一フォーマットで管理し、
最終的な報告書作成を効率化することを目的とする。

## 2. 前提・方針

* 日本語ユーザー向けに作る（画面文言・メッセージはすべて日本語）。
* 自分専用ツールとして利用する。共有機能・マルチユーザー機能は考えない。
* 最初はシンプルな構成にする。カスタマイズ項目は今後少しずつ増やす。
* 設定は誰にでも分かりやすいものにする（設定項目を増やしすぎない）。
* AGENTS.mdの方針（機密情報のマスク、攻撃自動化の禁止など）を踏襲する。
* 診断対象への自動攻撃・スキャン機能は実装しない。

## 3. 想定ユーザー

* このツールを作った本人（自分1人）。
* 学習・許可された検証環境での脆弱性診断メモ整理を目的とする。

## 4. 技術スタック

| 項目 | 選定 | 理由 |
|---|---|---|
| 言語 | Python | ユーザー指定 |
| Webフレームワーク | Flask | 軽量でシンプル。個人利用に十分 |
| テンプレートエンジン | Jinja2（Flask標準） | 自動HTMLエスケープが効くため、XSS対策の基本方針に合致 |
| データ保存 | SQLite | ユーザー指定。ファイル1つで完結し個人利用に十分 |
| ORM / DBアクセス | 標準の`sqlite3`モジュール | 学習コストを抑え、まずはシンプルに実装 |
| フロントエンド | 素のHTML + 自作の最小限CSS | 外部CDN・ビルドツールに依存せず、オフラインでも完全動作 |
| 認証 | なし | 自分専用のローカル利用が前提のため不要 |

外部LLM APIへの通信は初期バージョンでは行わない（詳細は9章参照）。

## 5. データモデル

診断は「案件（プロジェクト）」単位で行い、1つの案件の中に複数の脆弱性メモを記録する構成とする。

### 5-1. 案件（project）テーブル

| フィールド名 | 説明 |
|---|---|
| id | 案件ID（主キー、自動採番） |
| name | 案件名 |
| scope | 対象範囲（対象サイト・対象システムなど） |
| status | ステータス（進行中 / 完了） |
| created_at | 作成日時 |
| updated_at | 更新日時 |

### 5-2. 脆弱性メモ（memo）テーブル

AGENTS.md記載の項目に基づき、1件の脆弱性メモを以下の項目で管理する。

| フィールド名 | 説明 |
|---|---|
| id | メモID（主キー、自動採番） |
| project_id | 所属する案件ID（外部キー） |
| title | 脆弱性名（自由入力。SQLインジェクション・XSS等の候補を入力補助として表示） |
| target_url | 対象URL |
| http_method | HTTPメソッド |
| request_body | リクエスト内容（保存時にマスク済みの状態で格納） |
| response_body | レスポンス内容（保存時にマスク済みの状態で格納） |
| reproduction_steps | 再現手順 |
| observed_result | 確認結果 |
| impact | 影響 |
| severity | 深刻度（High / Medium / Low / Info の選択式） |
| severity_reason | 深刻度の判断理由（自由記述） |
| evidence_note | 証跡メモ（保存時にマスク済みの状態で格納） |
| fix_suggestion | 対策案 |
| status | 確認状況（未着手 / 調査中 / 確認済 / 報告済） |
| remarks | 備考（未確認事項、スクリーンショットの有無など） |
| created_at | 作成日時 |
| updated_at | 更新日時 |

* `severity` `status` `project.status` は選択式（プルダウン）にして入力揺れを防ぐ。
* `request_body` `response_body` `evidence_note` は、保存前に必ずマスク処理を通す
  （マスク方針は8章参照）。**マスク前の生データはDBに一切保持しない。**

## 7. 画面構成・URLルーティング（案）

| URL | 内容 |
|---|---|
| `/` | 案件一覧画面へリダイレクト |
| `/projects` | 案件一覧画面 |
| `/projects/new` | 案件新規作成画面 |
| `/projects/<id>` | 案件詳細＝配下メモ一覧画面（検索・絞り込み可） |
| `/projects/<id>/edit` | 案件編集画面 |
| `/projects/<id>/delete` | 案件削除（確認ダイアログ経由、配下メモもカスケード削除） |
| `/projects/<id>/export` | 案件単位のMarkdown一括エクスポート |
| `/projects/<id>/memos/new` | メモ新規作成画面 |
| `/memos/<id>` | メモ詳細画面（標準フォーマット表示） |
| `/memos/<id>/edit` | メモ編集画面 |
| `/memos/<id>/delete` | メモ削除（確認ダイアログ経由） |

メモ一覧画面（`/projects/<id>`）では、深刻度・確認状況によるプルダウン絞り込みと、
脆弱性名・メモ本文を対象にしたキーワードフリーワード検索を提供する。

## 8. マスク処理の方針

* AGENTS.md記載の対象（Cookie、セッションID、Authorizationヘッダー、パスワード、
  APIキー、メールアドレス、電話番号、個人名、住所、学籍番号・社員番号）を
  正規表現で自動検出する。
* メモの新規作成・編集フォームで、入力内容から自動検出したマスク候補箇所を
  ユーザーに提示し、確認・微調整できるようにする。
* 保存ボタン押下時に、マスク確定後のテキストのみをDBへ格納する
  （生データは保存しない。DBファイルが万一流出しても機密情報が残らないようにするため）。
* 表示・エクスポート時も、DBに格納されている時点で既にマスク済みのため、
  そのまま安全に表示・出力できる。
* 正規表現による自動検出には限界があるため、検出漏れは手動でユーザーが追記マスクできるようにする。

## 9. 「報告書用まとめ」とAI活用の関係

* AGENTS.mdに記載された2種類の情報構造は、以下のように整理する。
  * フォーム項目（脆弱性名／対象URL／HTTPメソッド／リクエスト／レスポンス／再現手順／
    影響／深刻度／証跡メモ／修正案／確認状況）＝ ユーザーが実際に入力する項目。
  * 標準フォーマット（概要／対象／再現手順／確認結果／影響／深刻度／対策案／備考）＝
    「報告書用まとめ」画面でフォーム項目を流し込んで表示するテンプレート。
* 「報告書用まとめ」画面では、上記テンプレートに沿ってフォーム入力内容を並べ替えて表示し、
  そのままMarkdownとしてコピー・エクスポートできるようにする。
* AGENTS.mdの「AIへの依頼時の期待動作」（貼り付けたメモから自動で重要情報抽出・
  脆弱性推定・影響・対策案をまとめる機能）は、**初期バージョンではアプリ機能として実装しない。**
  これは、ユーザーが本ツールを使用する際にAIコーディングエージェント（OpenCode等）へ
  相談する際の振る舞い方針、および `skills/` 配下の各スキルとして提供される。
* 将来的に、外部LLM APIと連携して「概要」「影響」「対策案」の下書きを自動生成する
  機能の追加を検討する（10章「今後の拡張候補」に記載）。

## 10. 主な機能（MVP：最初に作る範囲）

1. 案件の新規作成・一覧表示・編集・削除（確認ダイアログ付き、削除時は配下メモもカスケード削除）
2. 案件内メモの新規作成（フォーム入力 → マスク候補提示・確認 → マスク済みデータをSQLiteに保存）
3. メモの編集・削除（確認ダイアログ付き）
4. メモ一覧画面での深刻度・確認状況による絞り込み、キーワードフリーワード検索
5. メモ詳細表示（標準フォーマットで表示）
6. 入力値のHTMLエスケープ表示（XSS対策。Jinja2の自動エスケープを利用）
7. 案件単位でのMarkdown一括エクスポート（配下の全メモを標準フォーマットで連結）

## 11. 今後の拡張候補（MVP後）

memo.txtの「カスタマイズは今後増やしていく」という方針に沿って、
以下は後回しにする候補とする。

* PDF形式でのエクスポート
* 外部LLM APIと連携した「概要・影響・対策案」の自動下書き生成機能
* スクリーンショットなど画像ファイルの添付
* 脆弱性名のプリセットリスト（OWASP Top10等）の充実
* タグ・カテゴリ管理
* メモのテンプレートを複数用意して選べるようにする
* ダークモードなどの表示設定

## 12. セキュリティ・実装時の注意（AGENTS.mdより再掲）

* 入力値は表示時に必ずHTMLエスケープする（メモ本文にHTML/JSが含まれても実行させない）。
* Cookie、セッションID、Authorizationヘッダー、パスワード、APIキー、
  メールアドレス、電話番号、個人名、住所、学籍番号・社員番号は、
  保存時点でマスク済みのデータのみをDBに格納する。
* エクスポート機能はマスク済みデータのみを出力する。
* 診断対象への自動攻撃・スキャン機能は実装しない。
* ローカル保存（SQLiteファイル）であっても機密情報の扱いには注意し、
  `data/vulnnote.db` は `.gitignore` に追加してリポジトリに含めない。

## 13. 開発の進め方（フェーズ分け）

1. **フェーズ1：土台作り**
   Flaskアプリの雛形作成、SQLite初期化（案件・メモ両テーブル）、基本ルーティング確認。
2. **フェーズ2：案件のCRUD実装**
   案件の一覧・新規作成・編集・削除を一通り動かせるようにする。
3. **フェーズ3：メモのCRUD実装**
   案件に紐づくメモの一覧・新規作成・詳細・編集・削除を実装する。
4. **フェーズ4：マスク処理・エスケープ処理の実装**
   保存前のマスク候補検出・確認フロー、表示時エスケープを組み込み、動作確認する。
5. **フェーズ5：検索・絞り込み機能の実装**
   深刻度・確認状況によるプルダウン絞り込み、キーワード検索を実装する。
6. **フェーズ6：エクスポート機能**
   `skills/report-export-helper` の出力フォーマットに沿った、案件単位のMarkdown一括出力を実装する。
7. **フェーズ7：見た目・使い勝手の調整**
   自作シンプルCSSでの整形、入力しやすいフォームレイアウトの調整。
8. **フェーズ8以降：拡張機能**
   「11. 今後の拡張候補」から必要なものを順次追加する。

各フェーズ完了時に、AGENTS.md記載の通り
「入力・表示・保存・エクスポートの基本動作」を確認する。

## 14. ユーザーとの合意事項（決定の経緯）

`docs/grill-me.md` の指示に基づき、以下の点について一問一答形式で確認し、合意した。

| 論点 | 決定内容 |
|---|---|
| 実行環境・技術スタック | Python + Flask + SQLite |
| 認証機能 | なし（ローカル専用） |
| メモの管理単位 | 案件（プロジェクト）単位でグループ化 |
| 案件の項目 | 案件名・対象範囲・ステータス・作成日 |
| 機密情報マスクの実現方法 | 正規表現で自動検出＋ユーザーによる手動修正 |
| マスク適用タイミング | 保存時にマスク済みテキストへ変換してDB格納（生データは保持しない） |
| エクスポート形式 | Markdown（先行実装）。PDFは将来対応 |
| エクスポート単位 | 案件単位（配下の全メモをまとめて1つのMarkdownに） |
| 画像添付 | 初期バージョンでは非対応 |
| 確認状況の選択肢 | 未着手／調査中／確認済／報告済 |
| 検索・絞り込み | 深刻度・確認状況の絞り込み＋キーワードフリーワード検索 |
| 脆弱性名の入力方法 | 自由入力テキスト（候補入力支援付き） |
| フォームと報告書用まとめの関係 | フォーム項目を入力→「報告書用まとめ」欄で標準フォーマットへ変換表示 |
| 深刻度の入力方法 | 深刻度（選択式）＋理由（自由記述）の2項目 |
| AI自動抽出機能 | 初期バージョンでは非実装。将来的に外部LLM API連携を追加検討 |
| UIデザイン | 自作のシンプルCSSのみ（外部フレームワーク不使用） |
| 削除機能 | 案件・メモとも削除可能（確認ダイアログ付き、案件削除は配下メモもカスケード削除） |
