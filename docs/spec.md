| 項目 | 内容 |

## 2. プロジェクト概要

VulnNote Managerは、Webアプリケーション等の脆弱性診断中に生じる断片的なメモを、案件・対象・脆弱性の単位で整理するローカルWebアプリケーションである。診断中の記録、再確認、報告書作成までを一貫して支援する。

データは原則として利用者の端末内に保存する。OpenAI APIを使う機能だけは、利用者が内容を確認して明示的に実行した場合に限り、選択されたテキストを外部へ送信する。

## 3. 目的

- 診断メモを `案件 → 対象 → 脆弱性メモ` の階層で整理する。
- 再現手順、証跡、影響、対策を一貫した項目で記録する。
- 診断途中の断片的な状態でも保存でき、未記入項目を明確にする。
- 検索、絞り込み、並べ替えにより、過去の記録を再確認しやすくする。
- MarkdownおよびCSVへ安全に出力し、報告書作成へ再利用できるようにする。
- AIを文章整理と報告書の下書き作成に利用する。
- AIを含むどの機能でも、未確認事項を事実として補完しない。
- 機密情報の意図しない外部送信や、誤操作による削除を防ぐ。

## 4. 想定利用者・利用環境

- 利用者は脆弱性診断を行う個人とする。
- 初版は1人でのローカル利用を前提とし、ログイン認証は設けない。
- サーバーは既定でループバックアドレス `127.0.0.1` のみにバインドし、LANやインターネットへの公開を想定しない。
- Windows、macOS、Linuxを対象とする。
- Python 3.12以上の環境からコマンドで起動する。
- 対象ブラウザは、開発時点でサポートされているChrome、Edge、Firefox、Safariの最新版とする。

## 5. 初版の対象範囲

### 5.1 対象に含む機能

1. 案件の登録、閲覧、編集、削除、削除ロック
2. 対象の登録、閲覧、編集、削除、削除ロック
3. 脆弱性メモの登録、閲覧、編集、削除、削除ロック
4. 一覧表示、検索、絞り込み、並べ替え
5. 複数選択削除および表示中の全件を対象にした一括削除
6. スクリーンショットのアップロード、表示、並べ替え、説明文編集、削除
7. 個別メモおよび案件全体のMarkdown・CSV出力
8. OpenAI APIによるメモ本文の整理・文章校正
9. OpenAI APIによる診断報告書の下書き作成

### 5.2 初版の対象外

- 複数利用者、ユーザー管理、ログイン認証、権限管理
- インターネットまたはLAN上への公開運用
- 脆弱性メモの複製
- バックアップおよび復元（将来実装候補とする）
- Dockerによる起動・配布（将来実装候補とする）
- PDF、DOCX等への直接出力
- 外部のチケット管理、診断ツール、クラウドストレージとの連携
- モバイルアプリ

## 6. 用語

| 用語 | 定義 |
| --- | --- |
| 案件 | 1回の診断業務または診断プロジェクトを表す最上位の単位 |
| 対象 | 案件内で診断するWebサイト、Webアプリケーション、API等 |
| 脆弱性メモ | 1件の脆弱性または確認事項に関する記録 |
| 削除ロック | 対象データを誤って削除しないための保護状態。閲覧・編集は妨げない |
| 表示中の全件 | 現在の検索・絞り込み条件に一致する全件。画面上の1ページ分だけを意味しない |
| AI生成結果 | OpenAI APIが返した文章。利用者が採用するまでは確定情報として扱わない |

## 7. 機能仕様

### 7.1 案件管理

案件について、登録、詳細表示、編集、単一削除、複数選択削除、表示中の全件削除、削除ロックの切り替えを行えること。

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 案件名 | 必須 | 一覧および出力で案件を識別する名称 |
| 顧客名 | 任意 | 顧客または依頼元の名称 |
| 概要 | 任意 | 案件の目的、範囲、注意事項等 |
| 開始日 | 任意 | 診断開始日 |
| 終了日 | 任意 | 診断終了日。開始日より前の日付は登録不可 |
| 削除ロック | 必須 | 初期値は解除 |

### 7.2 対象管理

対象は必ず1つの案件に属する。登録、詳細表示、編集、単一削除、複数選択削除、表示中の全件削除、削除ロックの切り替えを行えること。

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 所属案件 | 必須 | 対象が属する案件 |
| 対象名 | 必須 | 対象を識別する名称 |
| ベースURL | 任意 | 対象の基準URL。URLとして妥当でない値には警告する |
| 概要 | 任意 | 対象範囲、除外範囲、注意事項等 |
| 削除ロック | 必須 | 初期値は解除 |

### 7.3 脆弱性メモ管理

脆弱性メモは必ず1つの対象に属する。登録、詳細表示、編集、単一削除、複数選択削除、表示中の全件削除、削除ロックの切り替えを行えること。

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 所属対象 | 必須 | メモが属する対象。案件は対象から特定する |
| タイトル | 必須 | 脆弱性を識別できる簡潔な名称 |
| 対象URL | 任意 | 脆弱性を確認した具体的なURL |
| 脆弱性の種類 | 任意 | 初期候補、過去の入力候補、または自由入力から指定 |
| 危険度 | 必須 | `Critical`、`High`、`Medium`、`Low`、`Info` のいずれか |
| 発見日時 | 必須 | 利用者のローカルタイムゾーンで入力・表示する |
| 概要 | 任意 | 現象や確認内容の要約 |
| 再現手順 | 任意 | 手順を順序立てて記録するテキスト |
| リクエスト・レスポンス | 任意 | HTTPリクエスト、レスポンス等の証跡 |
| 影響 | 任意 | 確認済みの影響。推測は推測と分かるように記載する |
| 対策方法 | 任意 | 推奨する修正・緩和策 |
| 対応状況 | 必須 | 7.4に定める値 |
| 削除ロック | 必須 | 初期値は解除 |

作成日時と更新日時はシステムが自動記録する。任意項目が空でも保存可能とし、詳細画面では未入力であることを表示する。入力内容をアプリが推測して補完してはならない。

### 7.4 対応状況

対応状況は次の7種類とする。

1. 未確認
2. 確認済み
3. 報告済み
4. 対応中
5. 修正済み
6. 再診断済み
7. 対象外

新規メモの初期値は `未確認` とする。

### 7.5 脆弱性の種類

初期候補は次のとおりとする。

- SQLインジェクション
- クロスサイトスクリプティング（XSS）
- クロスサイトリクエストフォージェリ（CSRF）
- OSコマンドインジェクション
- サーバーサイドリクエストフォージェリ（SSRF）
- パストラバーサル
- 認証不備
- 認可・アクセス制御不備
- セッション管理不備
- ファイルアップロード不備
- 情報漏えい
- セキュリティ設定不備
- オープンリダイレクト
- その他

候補にない種類は自由入力できる。過去に入力した値は次回以降の候補に表示する。比較時は前後の空白を除去し、完全に同じ表記の候補を重複表示しない。自由入力した値を自動で別の分類へ置き換えない。

### 7.6 一覧、検索、絞り込み、並べ替え

案件、対象、脆弱性メモはそれぞれ一覧表示する。脆弱性メモ一覧には、少なくともタイトル、案件、対象、対象URL、種類、危険度、発見日時、対応状況、更新日時、削除ロック状態を表示する。

案件一覧は、案件名、顧客名、概要を部分一致検索できること。案件名、顧客名、開始日、終了日、更新日時で昇順・降順に並べ替えできること。

対象一覧は、対象名、ベースURL、概要、案件名を部分一致検索できること。案件で絞り込み、案件、対象名、ベースURL、更新日時で昇順・降順に並べ替えできること。

脆弱性メモは次の項目を部分一致検索できること。

- タイトル
- 案件名、対象名
- 対象URL
- 脆弱性の種類
- 概要
- 再現手順
- リクエスト・レスポンス
- 影響
- 対策方法

次の条件で絞り込みできること。条件は併用できる。

- 案件
- 対象
- 危険度
- 脆弱性の種類
- 対応状況
- 削除ロック状態

次の項目で昇順・降順に並べ替えできること。

- タイトル
- 危険度（`Critical > High > Medium > Low > Info` の順序）
- 脆弱性の種類
- 発見日時
- 対応状況
- 作成日時
- 更新日時

対応状況を並べ替える場合は、7.4の記載順を意味順とする。昇順は `未確認` から `対象外`、降順はその逆順とする。

検索語と絞り込み条件は同時に適用する。検索結果が0件の場合は、データがないのか条件に一致しないのかを判断できるメッセージと、条件を解除する操作を表示する。

### 7.7 削除ロックと削除

案件、対象、脆弱性メモのすべてに削除ロックを設ける。ロックは削除だけを禁止し、閲覧および編集は可能とする。ロック状態の変更は、各詳細画面または一覧の明示的な操作から行う。

削除規則は次のとおりとする。

- ロックされた要素は削除できない。
- 案件を削除すると、配下の対象、脆弱性メモ、スクリーンショットも削除する。
- 対象を削除すると、配下の脆弱性メモとスクリーンショットも削除する。
- 自身または配下にロックされた要素が1件でもある案件・対象は削除しない。
- 単一削除でも確認画面を表示する。
- 親要素の削除前には、削除される案件、対象、脆弱性メモ、画像の件数を表示する。
- 複数選択削除では、削除可能な選択項目を削除し、ロック等により削除できない項目を残す。処理後に双方の件数と理由を表示する。
- 表示中の全件削除は、通常の複数選択削除とは異なる警告表示と再確認を必要とする。
- 削除処理はデータベースのトランザクションを用い、途中失敗で階層データが不整合にならないようにする。
- 初版にゴミ箱および復元機能は設けない。

### 7.8 スクリーンショット管理

1つの脆弱性メモへ複数画像をアップロードできること。

| 項目 | 仕様 |
| --- | --- |
| 対応形式 | PNG、JPEG、WebP |
| 上限 | 1ファイルにつき10MB |
| 保存名 | 元のファイル名を表示用に保持し、実ファイルは衝突やパストラバーサルを防ぐ安全な内部名で保存 |
| 説明文 | 画像ごとに任意の説明文を設定可能 |
| 表示順 | 利用者が並べ替え可能 |
| 削除 | 個別の確認後に、関連情報と実ファイルを削除 |

拡張子だけでなく、MIMEタイプとファイルシグネチャ（内容を識別する先頭部分）を検査する。許可形式以外、破損ファイル、上限超過ファイルは保存しない。画像は静的ファイル領域を直接公開せず、アプリケーションの画像配信用経路から適切なContent-Typeと安全なレスポンスヘッダーを付けて返す。

データベース登録とファイル保存のどちらかが失敗した場合に、片方だけが残らないように後処理する。メモまたは親要素の削除では、対応する画像ファイルも削除する。

### 7.9 Markdown出力

個別の脆弱性メモと案件全体をMarkdownで出力できること。

- 個別出力は、案件、対象、脆弱性メモの全項目を含む。
- 案件出力は、案件、対象、脆弱性メモの階層を見出しで表現する。
- 案件内の脆弱性メモは、危険度、対象、発見日時等で利用者が指定した順に並べられる。
- 未入力項目は `未記入` と明示し、推測で補完しない。
- リクエスト・レスポンス等は、内容がMarkdownやHTMLとして実行・解釈されにくい安全なコードブロックとして出力する。
- スクリーンショットを含む場合は、Markdownファイルと画像をZIPへまとめ、相対パスで参照する。
- ZIP内のファイル名を安全化し、絶対パスや `..` を含めない。

### 7.10 CSV出力

個別の脆弱性メモと案件全体をCSVで出力できること。案件全体では、1行を1件の脆弱性メモとする。

- 文字コードはUTF-8 BOM付きとする。
- 改行、カンマ、ダブルクォートを含む値を正しく引用する。
- 表計算ソフトで数式として解釈され得る先頭文字（`=`、`+`、`-`、`@`、タブ、復帰）を無害化し、CSVインジェクションを防ぐ。
- 任意項目の未入力値は空欄とする。
- スクリーンショットはバイナリを含めず、元のファイル名とアプリ内識別子を記録する。
- 出力列と順序は固定し、少なくとも案件、対象、脆弱性メモの全入力項目、作成日時、更新日時を含める。

### 7.11 AIによる文章整理・校正

利用者が選択した脆弱性メモについて、概要、再現手順、影響、対策方法等の文章をOpenAI APIで整理・校正できること。

処理手順は次のとおりとする。

1. 利用者がAI機能を明示的に起動する。
2. 外部送信される項目と内容を確認画面に表示する。
3. APIキー、Cookie、`Authorization` ヘッダー等の機密情報らしい文字列を検出した場合は警告する。
4. 利用者は送信前に内容を編集、マスキング、または送信中止できる。
5. 確認後、表示したテキストだけをOpenAI APIへ送信する。画像は初版では送信しない。
6. 原文とAI生成結果を比較表示する。
7. 利用者が採用する項目だけを選択して保存する。

AIには次の制約をプロンプトで明示する。

- 原文にない事実、検証結果、URL、手順、影響、対策を創作しない。
- 不明または根拠不足の内容は、断定せず `要確認` 等として明示する。
- HTTPリクエスト・レスポンス等の技術的な証跡を、校正目的で勝手に変更しない。
- 文章の整理と表現改善を行い、原文の意味を変えない。

生成結果は自動保存せず、利用者が採用した場合だけ元のメモへ反映する。

### 7.12 AIによる診断報告書の下書き作成

案件全体または利用者が選択した脆弱性メモから、Markdown形式の診断報告書下書きを生成できること。

- 送信対象の案件情報、対象情報、脆弱性メモを実行前に一覧・本文で確認できる。
- 送信対象から項目またはメモを除外できる。
- スクリーンショット画像は初版ではOpenAI APIへ送信しない。
- 下書きには、案件概要、対象、脆弱性一覧、各脆弱性の概要、再現手順、影響、対策、未確認事項を含める。
- 情報がない箇所を事実で補完せず、`未記入` または `要確認` とする。
- AI生成結果は元データを上書きしない。利用者が内容を確認した後にMarkdownとして保存・出力する。

### 7.13 AI設定とエラー処理

- OpenAI公式Python SDKとResponses APIを使用する。
- APIキーは環境変数 `OPENAI_API_KEY` から読み込み、画面、ログ、SQLite、エクスポート、バックアップ対象へ保存しない。
- モデル名は環境変数 `OPENAI_MODEL` で変更可能にする。
- 初期モデルは仕様策定時点でコストと品質のバランスを重視した `gpt-5.6-terra` とする。実装開始時に公式ドキュメントで提供状況を再確認し、利用不能または非推奨の場合は利用者確認のうえ本書を更新する。
- APIキー未設定、認証失敗、レート制限、通信失敗、タイムアウト、API応答不正を区別し、原因と対処を日本語で表示する。
- APIへ送信した本文や生成結果をアプリケーションログへ出力しない。
- AI機能の失敗によって、保存済みメモを変更しない。

### 8.1 画面一覧

| 画面 | 主な役割 |

| 案件登録・編集 | 案件情報の入力 |
| 案件詳細 | 案件情報、所属対象、所属メモ、案件出力、報告書下書き作成 |
| 対象一覧 | 対象の検索、案件絞り込み、並べ替え、選択、一括削除 |

| 対象詳細 | 対象情報と所属メモの確認 |

| 脆弱性メモ登録・編集 | メモ入力、スクリーンショット管理、AI校正 |
| 脆弱性メモ詳細 | 全項目と証跡の安全な表示、個別出力 |
| AI送信確認 | 外部送信内容の確認、編集、警告、送信同意 |
| AI結果確認 | 原文との差分確認、項目ごとの採用 |
| 削除確認・結果 | 削除件数、配下への影響、ロックによる除外、実行結果の表示 |

### 8.2 UI/UX方針

- 最低限の操作で目的を達成できる構成とする。
- Appleライクな余白、階層、角丸、穏やかな影、読みやすいタイポグラフィを参考にするが、Appleの商標や素材を模倣しない。
- 落ち着いた色調と親しみのある表現を用いる。
- 危険度は色だけに依存せず、必ず文字も併記する。
- 削除、外部送信、AI結果採用等の重要操作は、通常操作と視覚的に区別する。
- キーボード操作、フォーカス表示、ラベル、十分なコントラストを確保する。
- 長いリクエスト・レスポンスは等幅フォントで表示し、画面全体を崩さない。
- 保存成功、入力不備、処理中、失敗の状態を明確に表示する。
- エラーには、何が起きたか、データが保存されたか、利用者が次に何をすべきかを記載する。

## 9. データ仕様

### 9.1 エンティティと関係

```text
Project（案件） 1 ── * Target（対象） 1 ── * VulnerabilityNote（脆弱性メモ）
                                              1 ── * Screenshot（画像）
```

主要テーブルは次を想定する。物理名は実装時に英語のスネークケースで定義する。

| エンティティ | 主なシステム項目 |
| --- | --- |
| Project | ID、入力項目、削除ロック、作成日時、更新日時 |
| Target | ID、案件ID、入力項目、削除ロック、作成日時、更新日時 |
| VulnerabilityNote | ID、対象ID、入力項目、削除ロック、作成日時、更新日時 |
| Screenshot | ID、メモID、内部ファイル名、元ファイル名、MIMEタイプ、サイズ、説明文、表示順、作成日時 |

### 9.2 データ整合性

- SQLiteの外部キー制約を有効にする。
- 必須項目、列挙値、日付の前後関係をサーバー側で検証する。ブラウザ側の検証だけに依存しない。
- 日時はタイムゾーン情報を失わない形式で保存し、画面では利用者のローカル時刻として表示する。
- 更新競合を検出できるよう更新日時を利用し、古い画面からの更新で新しい内容を無言で上書きしない。
- データベーススキーマにはバージョンを持たせ、変更を再現可能なマイグレーションとして管理する。

### 9.3 保存場所

SQLiteファイルとアップロード画像は、ソースコードやGit管理対象とは分離したOS標準のユーザーデータ領域へ保存する。環境変数 `VULNNOTE_DATA_DIR` で保存先を変更可能にする。保存先が作成できない、書き込めない、容量不足である場合は、原因と確認先を日本語で表示して起動または保存を安全に中止する。

## 10. 技術仕様

### 10.1 採用技術

| 分類 | 技術 | 採用理由 |
| --- | --- | --- |
| 言語 | Python 3.12以上 | プロジェクト方針。型ヒントと標準ライブラリを活用できる |
| Web | Flask | 小規模なローカルWebアプリを少ない構成で実装できる |
| HTML生成 | Jinjaテンプレート | Flaskと組み合わせたサーバーサイドレンダリングに適する |
| 画面 | HTML、CSS、最小限のJavaScript | 初版の依存関係とビルド工程を抑える |
| データベース | SQLite、Python標準 `sqlite3` | DBサーバー不要で、検索・関連管理・トランザクションを利用できる |
| AI | OpenAI公式Python SDK、Responses API | 公式に保守されたSDKと現行APIを使用する |
| テスト | pytest | 読みやすいテストとフィクスチャを作成しやすい |

Flask、OpenAI SDK、pytest以外の外部ライブラリは、標準ライブラリで安全かつ明瞭に実装できない場合に限って追加する。追加時は目的、必要性、代替案を記録する。初版ではReact等のSPAフレームワーク、Node.jsのビルド環境、ORMを導入しない。

### 10.2 起動と設定

- Pythonパッケージとして構成し、最終的に `python -m vulnnote_manager` で起動できることを目標とする。
- 依存関係と対応Pythonバージョンは `pyproject.toml` で管理する。
- 開発用サーバーのデバッグモードを通常起動で有効にしない。
- ホスト、ポート、データ保存先、OpenAIモデル等の変更可能項目は環境変数または設定ファイルへ集約する。
- 秘密情報を設定ファイルのサンプル、ソースコード、Git履歴へ含めない。

## 11. セキュリティ・プライバシー要件

脆弱性診断記録には、認証情報、個人情報、未公開の脆弱性等が含まれ得る。そのため、ローカルアプリであっても次を必須とする。

- すべての表示値を既定でHTMLエスケープし、メモ内のHTMLやスクリプトを実行しない。
- リクエスト・レスポンスは安全なプレーンテキストとして表示する。
- データ変更操作にCSRF対策を行う。
- SQLはプレースホルダーを使用し、入力値をSQL文字列へ直接連結しない。
- URLやファイルパスの入力を、そのままサーバー側のファイル取得やコマンド実行に使用しない。
- アップロード要件は7.8に従い、パストラバーサル、危険なContent-Type、ファイル名衝突を防止する。
- エクスポート要件は7.9および7.10に従い、Markdown/HTML注入、CSVインジェクション、ZIP Slipを防止する。
- OpenAI APIへの送信は明示操作、送信内容確認、機密情報警告を必須とする。
- APIキーやメモ本文をログへ記録しない。例外メッセージにも秘密情報を含めない。
- ローカル利用という前提を画面またはREADMEに明示し、外部公開用の本番サーバーとして案内しない。
- HTTPレスポンスへ、用途に応じたCSP、`X-Content-Type-Options: nosniff` 等の安全なヘッダーを設定する。

削除ロックは誤操作防止機能であり、認証やアクセス制御の代替ではない。

## 12. 非機能要件

### 12.1 性能

- メモ1万件程度を保存した状態を初版の目安とする。
- ローカル環境で、通常の一覧表示、検索、絞り込みの95%を2秒以内に完了することを目標とする。
- 一覧はページ分割し、一度に全本文や全画像を読み込まない。
- 検索対象列と外部キーには、実測に基づき必要なインデックスを設定する。

### 12.2 信頼性

- 登録、編集、階層削除はトランザクションで処理する。
- 入力エラーやAI通信失敗で既存データを破損・上書きしない。
- 予期しないエラー時は、秘密情報を含まない診断情報をログへ残し、利用者には日本語で安全なメッセージを表示する。
- アプリ終了後もSQLiteと画像から状態を復元できること。

### 12.3 保守性

- Pythonの変数名、関数名、クラス名、モジュール名は英語とする。
- 型ヒントを使用し、表示、業務処理、データアクセス、外部API連携を分離する。
- OpenAI API連携をサービス層へ隔離し、メモ保存処理と密結合させない。
- 公開関数や複雑な処理には、目的と前提が分かる短いドキュメントを記載する。
- エラーメッセージは、原因と利用者が取れる対処を日本語で示す。

### 12.4 アクセシビリティ

- フォーム入力には視覚的なラベルを付ける。
- キーボードだけで主要操作を実行できる。
- フォーカス位置を視認できる。
- 色だけで危険度、状態、エラーを表現しない。
- 可能な範囲でWCAG 2.2 AA相当のコントラストと操作性を目標とする。

## 13. 開発方針

- 小さな単位で実装し、各機能に自動テストを追加する。
- 標準ライブラリで十分な場合は外部依存を追加しない。
- 既存の設定、命名、ディレクトリ構成が追加された場合は、それらを優先する。
- AIエージェントを開発補助に使用するが、生成コード、仕様適合性、セキュリティ、テスト結果は人が確認する。
- AIエージェントは未確認の要件を推測で確定しない。仕様変更が必要な場合は、利用者への確認と本書の更新を先に行う。
- セキュリティに関係する入力、ファイル、エクスポート、AI送信、削除処理を重点的にレビューする。
- 変更理由と操作内容を、初学者でも追える具体的な日本語で記録する。

## 14. テスト方針

少なくとも次を自動テストの対象とする。

- 案件、対象、脆弱性メモの正常登録・編集と必須入力検証
- 日付前後関係、危険度、対応状況等の不正値拒否
- 階層関係とSQLite外部キー制約
- 検索、複合絞り込み、全並べ替え条件
- 単一削除、複数選択削除、表示中の全件削除
- 自身または配下の削除ロックによる削除拒否
- 親削除時の配下データ・画像削除と、失敗時のロールバック
- 許可画像のアップロード、サイズ上限、偽装拡張子、不正ファイル名の拒否
- HTMLエスケープ、CSRF、SQLインジェクション対策
- Markdown、CSV、ZIP出力と各種インジェクション対策
- OpenAI API送信前確認、警告、送信取消、APIエラー、結果の不採用
- AI生成結果が原文を自動上書きしないこと
- APIキーやメモ本文がログに出ないこと

OpenAI APIを呼ぶ単体・結合テストでは通常モックを使用し、実APIを使うテストは明示的に分離する。

## 15. 初版の受け入れ条件

初版は、次の条件をすべて満たしたとき完成とみなす。

1. Python 3.12以上のWindows、macOS、Linuxで手順どおり起動できる。
2. 案件、対象、脆弱性メモを登録・閲覧・編集・削除できる。
3. 診断途中のメモを任意項目未入力で保存でき、未入力が明示される。
4. 危険度、種類、対応状況等で検索・絞り込み・並べ替えできる。
5. 3階層すべてで複数選択削除と表示中の全件削除を実行できる。
6. 削除ロックされた要素と、その要素を配下に持つ親が削除されない。
7. PNG、JPEG、WebPを1件10MB以内で複数添付し、安全に表示・削除できる。
8. 個別メモと案件全体をMarkdownおよびCSVで安全に出力できる。
9. AIへ送る内容を事前確認でき、機密情報らしい文字列には警告が表示される。
10. AIの文章整理結果を比較し、選択した内容だけを採用できる。
11. AIが作成した報告書下書きを確認後にMarkdownとして保存・出力できる。
12. AI処理が未確認事項を事実として自動保存せず、API失敗時にも保存済みデータが変化しない。
13. 本書のテスト方針に対応する自動テストが成功する。
